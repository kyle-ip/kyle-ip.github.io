什么是 API 网关？API 的简单性和稳定性

在 微服务) 架构中，随着服务及其接口的激增，API 网关应运而生。其核心目的是简化和稳定对客户端暴露的接口。除此之外，由于 API 网关在架构中的独特位置，可在其上实现多种附加功能，例如监视，日志，安全，负载均衡和流控。

The decline of Heroku | InfoWorld

API网关类似于外观设计模式，但却适用于网络级别。在两种情况下，其目的都是提供一种简化的外观，以隐藏系统的复杂性。

简化接口

PI网关的作用是使客户端与服务解耦，从而为入站和出站流量提供单点联系。

微服务后端可能涉及许多交互服务，并且其 URL 和协议可能是异构的。API 网关可以为客户端提供简化的界面来交互。某种意义上就像简化代理。

另外，API 网关可能更复杂，可接收单个请求，检索必要的资源并将其组合为单个响应。例如，对用户信息的请求可能会检索用户详情，包括最新的消息和其他感兴趣内容。网关可以接受单个请求，从必要相关的服务中获取数据，将它们整合为单个响应。

这种功能需要在架构层面进行编排，而且还会对熔断器和服务网格等其他微服务组件有影响。

由于 API 网关可以转换后端使用的协议和URL，它在服务迁移方面很有用，也就是说在某种程度上 API 网关可隐藏后端发生变更。

API 稳定性

随着服务开发进行，API 网关还可以提高与客户端交互的 API 的稳定性。网关可消除可能导致客户端中断或需要更改的细微变更，开发人员可使用网关将某些流量定向到更新的服务版本以对其进行测试。 通常 API 网关为客户端提供单点联系，这有助于尽可能减小后端服务变更带来的影响。 服务越复杂，变更频率越高，网关的功能就越有价值。

需要注意的是，即使对于内部服务 API 网关也可以用来简化 API，也就是说可以使用网关来简化微服务架构拓扑的管理。

监控

在架构中 API 网关还是监控，告警和跟踪的理想角色。同样的，在网关处收集统计信息对于流量和使用情况的广泛分析很有用。

有多种方法可用于分析 API 网关日志记录，例如使用 Amazon Elasticsearch Service。

计费与限流

如果按使用量付费使用 API，则 API 网关则很适合用于费率限制或跟踪费用。在一个单点跟踪客户和 API 的使用，要比尝试检测各种服务以达到相同的效果要简单得多。

安全

由于网关是进入系统的单点入口，所以它可以在最前面对系统进行加固。这类似于跳板机在系统管理最佳实践中的角色。

服务可以只着眼于与 API 网关的通信安全，比处理多个面向公众的 API 更为简单。

以上做法适用于基础架构级别的安全。而网关也可以在业务级别上承担身份验证和授权的工作。这需要对所涉及到的后端服务进行编排，但可以通过将安全逻辑集中在一个地方来简化工作。这自然会与限流或按需计费挂钩，因为网关将持有用于计费的客户帐户。

解决方案

API 网关解决方案分为两大类：托管和自行管理。

在托管领域，由每个云供应商都提供产品。可搜索“Amazon API Gateway, “Azure Application Gateway,” and “Google Cloud API Gateway.”。

而当要自己管理 API 网关时，最好从一个框架开始，由该框架帮助完成诸多繁重的工作。例如在 Java 生态中 Spring Cloud 提供了 Spring Cloud Gateway，而 Nginx 也是另一个不错的选择。

API 网关实践

我们已经讨论了 API 网关带来的许多好处，但务必要记住一个折衷方案：网关是一项需要维护的基础设施，特别是必须与后端、客户端的变更保持同步。也就是说如果架构复杂性或业务案例可以考虑它，那么 API 网关将是理想的解决方案。
