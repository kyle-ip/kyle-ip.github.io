关于数据模型：

  • 复杂的应用程序可能会有更多的中间层，其基本思想：每层都通过提供一个简洁的数据模型隐藏下层复杂性。抽象机制使得不同的人群可以高效协作。

  • 有许多不同类型的数据模型，每种数据模型都有其最佳实践。精通一种数据模型需要很大功夫（即使不关心内部机制）。由于对其上的软件应用有巨大影响，因此需要慎重选择合适的数据模型。

数据模型

关系模型：严格关联

在关系模型中数据被组织成关系（Repations），在 SQL 中被称为表（Table）,每个关系都是元组（Tuples）或无序集合（行），其实现为关系数据库管理系统（RDBMS）和 SQL。

数据访问

  • 关系模型定义了所有数据的格式：关系（表）只是元组（行）的集合，没有复杂的嵌套结构和访问路径，可以读取表中的任何一行或者所有行，支持任意条件查询。

  • 可指定某些列作为键并匹配这些列来读取特定行。

  • 可在任何表中插入新行，而不必担心与其他表之间的外键关系。

  • 优势在于联结操作、多对一和多对多关系更简洁的表达上，与文档模型抗衡。

  • 查询优化器：只需构建一次，使用该数据库的所有应用程序都可以从中受益。即由查询优化器决定以何种顺序执行查询、使用哪些索引。希望使用新的方式查询数据，只需声明一个新的索引，查询会自动使用最合适的索引。

局限性

  • 数据存储在关系表中，应用层代码中的对象与表、行和列的数据库模型之间需要一个笨拙的转换层。模型之间的脱离有时被称为阻抗失谐。使用 ORM 框架可减少转换层所需的样板代码量， 但也不能完全隐藏两个模型之间的差异。

  • 关系型数据库遵顼严格的 写时模式，因此在数据量大时，模式更改、数据更新操作速度缓慢且可能需要停机。此时可以借鉴文档数据库的做法采用 读时填充：设置字段为 NULL，在读取时再填充值。

多对一关系

通常将诸如地址、行业等定义为 id（设计为字典表），而字符串在填写时供用户选择（而非输入），好处在于：

  • 保持样式和输入值一致

  • 避免歧义（广州、广州市）

  • 易于更新

  • 本地化支持

  • 搜索支持

多对一 关系的表达（多人生活于某个地区、从事某一行业）不适宜使用文档模型：

  • 区别于一对多的树状结构（不需要联结，即使支持联结通常也很弱），相关项都由唯一的标识符引用，在关系模型中被称为外键，在文档模型中被称为文档引用。标识符可以查询时通过联结操作或相关后续查询来解析。迄今为止文档数据库并未遵循 CODASYL 标准。

  • 如果数据库本身不支持联结，也必须在应用程序代码中通过对数据库进行多次查询来模拟联结，即从持久层转移到应用层。

  • 即使应用程序初始版本适合采用无联结的文档模型，随着应用支持越来越多的功能，数据也变得更加互联一体化。

多对多关系

非关系模型

此处主要指 NoSQL，其目的不是替代 SQL，而是“不仅仅是 SQL”。采用 NoSQL 数据库有几个驱动因素，包括：

  • 比关系型数据库更好的扩展性需求。

  • 丰富的免费和开源软件。

  • 满足特定需求的查询操作。

  • 更少限制、更具动态和表达力的数据模型（不会对存储的数据强加模式）。

目标是通过提供新的数据模型以及内置的复制和分区等手段改进传统关系模型。NoSQL 有时也会结合关系型存储一起使用，被称为混合持久化。常见用于解决关系型数据库问题的 NoSQL 方案：

  • 键值数据库：解决无法存储数据结构的问题（Redis）。

  • 文档数据库：解决强 schema 约束的问题（MongoDB）。

  • 列式数据库：解决大数据场景下的 I/O 问题（HBase）。

  • 全文搜索引擎：解决全文搜索性能问题（Elasticsearch）。

文档模型：弱关联

支持文档数据模型的主要论点是模式灵活性， 由于局部性而带来较好的性能，对于某些应用而言它更接近于应用程序所使用的数据结构。

目标用例是数据来自于自包含文档，且一个文档与其他文档之间的关联很少。

灵活性

文档数据库是某种方式的层次模型：即在其父记录中保存了嵌套记录，而不是存储在单独的表中。

  • 读时模式：即为读数据的代码通常采用某种结构因而存在某种隐形模式（而不是由数据库强制执行，类似运行时类型检查），适用于大量异构、动态结构的数据存储。而关系型数据库的 写时模式 在更改模式时则面临速度慢和停机的问题（存在于数据量大的情况下，有各种工具可以解决限制）。

  • 非关联的树状结构也很容易表达 一对多 关系，比如一名用户身兼多个职位、拥有多段工作经历等。

  • 当应用数据在一对多关系中，通常一次加载整个树，使用文档模型更合适。因为无需像关系模型般把文档结构分解为多个表，使得模式更为笨重，以及不必要的应用代码复杂化。

  • 文档模型局限性在于不能直接引用嵌套项，难以处理嵌套太深的结构。对于联结的支持（多对多、多对一）缺失是否是问题取决于应用程序。

局限性

  • 在表示多对一和多对多的关系时， 关系数据库和文档数据库没有根本的不同：在这两种情况下相关项都由唯一标识符引用（称为外键或文档引用）。可以查询时通过联结操作或相关后续查询来解析（文档数据库并未遵循 CODASYL 标准）。

  • 如果应用程序使用多对多关系，文档模型可以通过反规范化减少联结需求，但应用程序代码需要做额外的工作来保持非规范化数据的一致性，除了代码更复杂外通常也比数据库内的专用代码执行的联结慢。

数据局部性

由于文档通常存储为 JSON、 XML 或其二进制变体（如 MongoDB 的 BSON）编码的连续字符串：

  • 存储局部性具有性能优势（相比起关系模型数据被划分到多个表，需要多次索引查找来检索，中间需要更多的磁盘 I/O）。

  • 仅适用 需要同时访问文档大部分内容 的场景。如果只是访问其中的一小部分，对于大型文档数据则比较浪费。

  • 对文档进行更新时通常会重写整个文档，而只有修改量不改变源文档大小时原地覆盖更新才更有效。因此建议文档尽量小且避免写入时增加文档大小。

图模型：复杂关联

关系模型能够处理简单的多对多关系，随着数据之间的关联越来越复杂，将数据建模转化为图模型会更加自然（目标用例是所有数据都可能会互相关联）。

  • 由顶点和边组成。每个顶点可以连接到其他任何顶点，与文档模型类似，不存在模式限制某种事物是否可关联。给定某个顶点可高效地得到所有入边和出边，从而遍历图。

  • 很多著名算法可以在图上运行，如 PageRank。

  • 除了同构数据，还提供了单个数据存储区中保存完全不同类型对象的一致性方式。通过对不同类型的关系使用不同的标签，可以在单个图中存储多种不同类型的信息，同时仍然保持整洁的数据模型。

图相关模型和处理框架：

  • 属性图模型：比如 Neo4j 、 Titan 和 InfiniteGraph。

  • 三元存储模型：比如 Datomic、 AllegroGraph。

  • 图处理框架：比如 Pregel。

数据查询语言

声明式语言

  • 命令式语言：如数据处理代码，完全可以推理整个过程， 逐行遍历代码、 评估相关条件、更新对应的变量，并决定是否再循环一遍。由于指定了特定的执行顺序而难以并行化

  • 声明式语言：如 SQL 与关系代数，只需指定所需的数据模式， 结果需满足什么条件，以及如何转换数据，而不需指明如何实现这一目标。SQL 隐藏了数据库引擎的细节，数据库系统能在不改变查询语旬的情况下提高性能，且通常适合于并行执行。

MapReduce

MapReduce 是一种编程模型，用于在许多机器上批量处理海量数据。

  • 介于声明式查询语言和命令式查询 API 之间：查询的逻辑用代码片段来表示，代码片段可以被处理框架重复地调用。

  • map 和 reduce 函数对于可执行的操作限制必须为纯函数，只能使用传递进去的数据作为输入，不能执行额外的数据库查询，也不能有任何副作用。使得数据库能够在任何位置、 以任意顺序来运行函数，并在失败时重新运行这些函数。

  • 某些 SQL 数据库也可以扩展使用 JavaScript 函数，在 MapReduce 模型中一个可用性问题是必须编写两个密切协调的JavaScript 函数，通常比编写单个查询更难。

  • 聚合管道：声明式查询语言的支持，表达能力上相当干 SQL 的子集，但它使用了基于 JSON 的语法。

图查询语言

声明式：

  • Cypher：用于属性图的声明式查询语言，最早为 Neo4j 图形数据库而创建。

  • SPARQL：采用 RDF 数据模型的三元存储查询语言。

  • Datalog

  • SQL：在关系数据库中，通常会预先知道查询中需要哪些 join 操作。而对于图查询，在找到要查找的顶点之前可能需要遍历数量未知的边，join 操作数量并不是预先确定的。因此在图模型中使用 SQL 会非常繁琐。

命令式：Gremlin
