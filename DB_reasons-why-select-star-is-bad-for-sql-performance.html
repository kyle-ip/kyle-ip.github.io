SELECT * 对 SQL 性能不利的原因

假设应用程序不需要访问表中的所有列，则使用 SELECT * 是不利于 SQL 性能，下文将给出具体原因。在编写生产代码的时候，不仅出于性能方面的考量，还会为了确保应用程序可靠在 select-list（映射）中明确指定关注的列。例如当添加新列或表中的列顺序发生更改时，应用程序的数据处理程序是否会突然崩溃。本文仅关注 SQL 性能，使用 Oracle 作为例证，而大多数的推理也适用于其它现代关系型数据库。

增大网络流量开销

这是最明显的效果。如果从每行返回 800 列而不是 8 列，每次查询执行都会在网络上发送 100 倍以上的数据量（实际开销可能会因各列的长度不同而有所不同）。网络上传输更多字节意味着发送了更多的数据包，并且取决于 RDBMS 实现，也意味着应用程序与数据库之间有更多的网络往返。

在突发的请求中，Oracle 可以将多个连续 SQL * Net 数据包中的单个访存调用结果数据，以流式传输到客户端，而无需客户端应用程序逐个确认先前的数据包。这种突发的吞吐量取决于 TCP 发送缓冲区的大小，以及网络链路带宽和延迟（参考 SQL*Net message to client vs SQL*Net more data to client | Tanel Poder Consulting）。

其花费了约 1m 35s 的时间，将 169 MB 的数据从数据库发送到客户端（平均每行大约 100 个字节）。根据数据字典统计数据粗略估算的行长度表明，平均行大小应为 119 bytes（116 再加上 3 bytes 的行头部，其中包括 lock byte 和列数）：

上表只有 16 列，如果仅选择应用程序所需的 3 列：

从 16 列中仅选择 3 列可使查询响应时间缩短为一半以下（1m 35s vs 43 sec）。 sqlplus 的度量标准包括在数据库服务器上执行查询的时间，以及从客户端获取所有记录所花费的时间。因此网络延迟，吞吐量和 TCP 发送缓冲区配置将对其产生影响。

Oracle 可以删除每个访存调用结果集中的重复字段值，因此如果需要访存大量行和列，为了节省网络带宽（例如将数据从纽约复制到新加坡），则可以通过利用同样宽的、最重复（即最多不相同的值）列对查询结果集进行排序，来最大化实现这种“压缩”。

上面的测试再次使用 SELECT *，并根据几个 VARCHAR2 类型的列进行排序，这些列的大小为 10-40 字节，并且具有很多重复的值。

在服务器进行 SQL * Net 协议级重复数据删除后，仅发送了约 65 MB。 需要注意的是上述所有测试运行的，往返于客户端的 SQL * Net 值都是相同的。这是因为在应用程序中，访存 arraysize 参数被设置为 100。arraysize 参数控制最终通过网络发送、以进行数据检索的 fetch 调用的数量。无论列的宽度如何，第一个请求之后的每次 fetch 都会请求返回 arraysize-ful 的行：

因此 SQL * net 往返度量标准取决于通过网络发送的数据库调用次数（fetch），而每个往返衍生度量发送的字节数，取决于获取单个数据库调用中请求的行数以及这些行的宽度。 实际情况稍微有点复杂，取决于应用程序客户端库的行为（略）。

需要注意的是，如果进一步增加 arraysize（例如增加到 1000），不仅会减少 SQL * Net 往返次数（1700 -> 16994，而不是 SQL * Net往返次数），还会略微减少传输的字节数（可能是由于更好压缩和稍低的 SQL * Net 数据包开销）。在 Oracle 数据库链接传输数据时，无需在客户端会话中增加 arraysize，因为 Oracle 自动为链接使用最大的 arraysize（〜32767）。

增大客户端 CPU 使用率

在客户端上处理的行越多，所拥有的列越多/ 越宽，处理所花费的 CPU 时间就越多。在应用程序中，花费的时间与提取、格式化记录并将它们写入文件这些操作有关。

某些查询计划优化无法执行

Oracle 的查询优化器可以将查询结构转换为其他结构以提高性能，其结果在逻辑上是等效的。 有些转换带来了更多的优化机会（更有效的数据访问路径），有些甚至允许跳过部分查询的执行。

例如有一个索引覆盖了待查询的所有列，则 Oracle 可以仅对体积较小的索引（而不是大表）进行扫描。 这种快速完整扫描不使用索引树遍历，而更像是对所有索引块按其存储顺序进行的全表扫描（忽略根块和分支块）。
