作为 NoSql 数据库的 Git

Git 的官方手册指出，它是一个愚蠢的内容跟踪器。Git 可能是世界上使用最广泛的版本控制系统，奇怪的是它没有将自己描述为源代码控制系统。

而实际上使用 Git 可以跟踪任何类型的内容，例如创建一个基于 Git 的 NoSQL 数据库。

官方手册说它“愚蠢”的原因是 Git 不对用户存储的内容做任何假设，底层的 Git 模型是相当基本的。本文打算探讨将 Git 用作 NoSQL 数据库（键值存储）的可能性，即可以将文件系统用作数据存储，然后使用 git add 和 git commit 来保存文件：

这样做效果很好，但现在这种做法是把文件系统作为数据库：路径是键，值是存储在其中的任何东西，其中有几个弊端：

  • 需要把所有的数据写到磁盘上，然后才能把它们保存到 Git 中。
  • 需要多次保存数据。
  • 文件存储没有被重复利用，失去 Git 提供的自动删除重复数据的优势。
  • 如果想同时在多个分支上工作，需要切换、创建多个目录。

目标是得到一个裸仓库，其中没有任何文件存在于文件系统中，而只存在于 Git 数据库中。

以下是 Git 的数据模型及其工作的管道命令。

现在讨论将 Git 用作 NoSQL 数据库的潜在优势和缺点。

数据效率

在存储数据方面，Git 是非常高效的。如前所述，由于哈希值的计算方式，相同内容的 blobs 只会被存储一次。可以在一个空的 Git 仓库中添加一大堆内容相同的文件，然后检查 .git 目录的大小与磁盘上的大小，可以发现 .git 目录小了不少。

不止于此，Git 对树也有同样的作用。如果在一棵子树上修改了一个文件，Git 只会创建一个新的子树，而只是引用其他没有受到影响的树。

尽管 Git 在引用已有数据方面非常高效，但如果每一个小的修改都生成一个完整的拷贝，在一段时间后还是会得到一个巨大的版本库。

为了缓解以上问题，引入了自动垃圾收集过程。git gc 命令运行时，它将查看所有的 Blobs，并在可能情况下删除 Blobs，转而存储基本数据的单一副本，以及 Blobs 每个版本的增量。因此 Git 仍然可以检索到 blob 的每个唯一版本，而不需要多次存储数据。

Git 工具

Git 拥有丰富的工具集，可用于检索和处理数据。虽然大多数记录都集中在代码上，但并不意味着不能用它们来处理其他数据。

基本的 Git 命令：

  • 使用 git diff 查找两个提交/ 分支/ 标签/ 的变化差异。
  • 使用 git bisect 找出由于数据变化而停止工作的情况。
  • 使用 git hooks 获取自动变更通知，建立全文索引，更新缓存，发布数据，…
  • 还原、分支、合并，…

还有一些外部工具：

  • 使用 Git 客户端实现数据可视化以便浏览 。
  • 使用 pull requests 在合并之前检查数据更改。
  • Gitinspector：Git 存储库的统计分析。

NoSQL

由于它是一个键值存储，Git 具备一般 NoSQL 存储的优势：比如无模式的数据库，可以存储任何内容，它甚至不一定是 JSON。

连接性

Git 可以在分区网络中工作，可以将所有内容放在 USB 存储上，在不连接网络时保存数据，然后在重新联机时推送并合并。这在开发代码时的常用做法相同。在某些场景中这可以节省时间。

事务

上面的例子是提交了对一个文件的每一个改动，但有时不一定要这样做，还可以把各种变化作为一个提交，这样做可以方便以后原子化地回滚这些改动。

长时间的事务：创建一个分支，向其提交几个改动然后合并或丢弃它。

备份与副本

对于传统的数据库，通常要为完整的备份和增量的备份建立一个调度计划，但这是相当麻烦的。由于 Git 已经存储了所有历史记录，所以永远不需要做完整的备份。

此外备份仅仅是执行 Git 推送，而可以推送到任何地方，GitHub、BitBucket 或自我托管的 Git 服务器。

上述是 Git 作为数据库的优点，然而还有以下缺点。

查询

只支持按键查询，唯一的好处是可以将数据在目录中组织，这样就可以轻易地通过前缀获得内容。任何其他的查询都是不支持的，除非执行完整的递归搜索。

唯一的做法是建立专门用于查询的索引。如果不需要过时，可以按调度计划维护索引，或者可以在提交发生后立即使用 git hook 更新索引。

并发

如果只是在编写 Blob，那么不必担心并发问题。但当开始编写提交并更新分支时，就会出现问题了。

下图中两个进程同时尝试创建提交：

在上图中可见，当第二个进程修改树的副本时，它实际上是在一棵过期的树上工作。当提交时它将失去第一个进程所做的修改。

同样的情况也适用于移动分支头。在提交和更新分支头之间可能会有另一个提交，因此有可能将分支头更新为错误的提交。这种问题唯一的解决方法是锁定从读取当前树副本到更新分支头之间的任何写操作。

速度

在创建分支时 Git 的速度很快，但当考虑到每秒提交次数时，它实际上就很慢了，因为需要一直向磁盘写入。

通常在写代码时不会注意到这点，因为每秒不会做很多提交。

在本地机器上进行一些测试，可见极限时每秒 110 次。虽然 110 次/秒的提交量对很多应用来说是足够的，但不能满足所有的情况。这是本地开发机器上的理论最大值，而其他因素都会影响速度。

树的大小

一般来说，用户倾向于使用很多子目录，而不是把所有的文件放在同一个目录中，这将使写入速度尽可能地接近最大值。原因是每次创建一个新的提交时都必须复制树，再对其进行修改，然后保存修改后的树。运行 git gc 会确保将其保存为一个 delta，而不是两个不同的树。

在第一种情况下，根目录下存储有 10.000 个 blobs。当添加一个文件时会复制包含 10.000 个项目的树，添加一个并保存它。由于树的大小，这可能是一个冗长的操作。

在第二个示例下有 4 层树，每层有 10 个子树，最后一层有 10 个 blobs（1010*10 \ 10 = 10.000 个文件）。

在这种情况下，如果想添加一个 blob，不需要复制整个层次结构，而只复制引导 blob 的分支（下图显示了必须要复制和修改的树）。

因此通过使用子目录，可以复制 5 棵有 10 个条目的树，这将提高了速度。而当数据增长得越多，就越希望使用子目录。

在事务中合并值

如果需要以每秒 100 次以上的速度进行提交，那么也许不必地单独回滚。在这种情况下，可以不提交每个改动，而是在一次提交中合并提交多个改动。由于可以同时写 Blobs，所以可以同时写 1000 个文件到磁盘，然后做一次提交，把它们保存到版本库中。但这也有缺点，如果你想获得原始的速度，这是个好办法。

解决这个问题的方法是给 git增加一个不同的后端，它不会立即把内容刷到磁盘上，而是先写到内存数据库中，然后再异步刷到磁盘。不过，实现这一点并不那么容易。当我使用libgit2sharp连接到版本库来测试这个解决方案时，我尝试使用Voron后端（它可以作为开源版本，以及使用ElasticSearch的变体）。这使速度提高了不少，但你失去了能用任何标准的git工具检查数据的好处。
