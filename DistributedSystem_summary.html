分布式系统知识点总结

协调与同步

分布式互斥

集中式 分布式 令牌环
方法 引入协调者，对请求排序，最早请求的参与者取得临界资源。 征求其它参与者同意后使用临界资源。 参与者组成一个环，轮流使用资源（不需要使用就传递给下一位）。
优点 简单，通信高效，易于实现。 可用性高。 可用性高，单个参与者通信效率高。
缺点 可用性低，协调者是性能瓶颈（单点）。 通信成本高，复杂度高。 当参与者对资源使用频率较低，带来较多无用通信。
场景 协调者可靠性和性能有一定保障。 临界资源使用频度较低且系统规模较小 系统规模较小，且每个参与者使用资源频度较高、时间较短。

分布式选举

Bully Raft ZAB
选举原则 节点 ID 最大的成为 Leader。 少数服从多数，票数最多为主。 倾向于让数据最新或 ID 最大的节点成为主。
优点 容易理解，选举速度快，算法复杂度低，易于实现。 选举速度快，算法复杂度低，易于实现。 性能高，对系统无特殊要求。
缺点 信息存储量大（每个节点存储所有节点信息），易频繁换主，不适用于大规模系统。 要求系统全连接，消息通信量大，不适用于大规模系统。 选举时间长，复杂度高。
场景 小规模系统，比如 MongoDB。 中小规模系统，比如 Kubernetes。 大规模系统，比如 ZooKeeper。

分布式共识

PoW PoS DPos
原理 以每个节点计算能力竞争记账权，算力越高机会越大。 由系统权益决定区块记账权，权益越大机会越大。 持有币则可进行投票，选举出节点作为代表记账。
优点 相对公平，有容错机制，完全去中心化，简单易于实现。 资源消耗低，达成共识周期短，交易服务费低。 能耗更低，每秒完成交易量高，无垄断，交易服务费低，更安全。
缺点 不适合私有链或联盟链，共识效率低、每秒完成交易量少，存在阻塞问题，资源浪费严重，交易服务费高。 每秒完成交易量较低，容易被垄断，无法处理分叉链情况。 持币者投票积极性不高，故障解决效率低，易出现安全隐患。
场景 比特币 以太坊、点点币 以太股、EOS

分布式事务

2PC 3PC 最终一致性
思想 事务管理器作为协调者，处理各本地事务的提交或回滚。资源管理器作为参与者。通过投票阶段和提交阶段协调事务操作。 在 2PC 基础上，分为 CanCommit、PreCommit、DoCommit 三个阶段，引入超时和准备机制。 将事务以保存在本地文件、数据库或消息队列中的消息或日志异步执行，通过业务规则进行重试。
特点 强一致性，同步执行，简单易实现。 强一致性，同步执行，简单易实现。相对 2PC 而言缓解了阻塞和单点故障问题。 最终一致性，异步执行，性能高，无阻塞和单点故障问题。
缺点 同步阻塞，单点故障，数据不一致，性能低。 数据不一致，性能较低。 算法复杂度高，实现困难。

分布式锁

关系型数据库 Redis ZooKeeper
实现 在表上对临界资源做唯一性约束，通过增加记录对某资源上锁，释放时删除（也可以更新实现乐观锁）。 使用 API setnx，设置锁 ID、当前时间 + 超时时间。 在持久节点 shared_lock 目录下为每个进程创建临时顺序节点，创建最小编号的节点的进程获得锁，否则等待较小的编号节点释放锁。
优点 容易理解、实现。 性能高，可跨集群部署、无单点故障，易于实现。 无单点故障，可重入，无死锁，可靠性高。
缺点 单点故障，性能低，可靠性低。 锁失效时间控制不稳定，可靠性低，较复杂。 性能较低，实现复杂。
场景 并发量低、性能要求低。 高并发、性能要求高。 大部分分布式场景，除非性能要求极高。

管理与调度

体系结构

集中式 非集中式
简介 一或多台中央服务器，存储系统内的所有数据、处理所有业务。中央服务器统一调度资源和任务，多个节点与中央服务器连接，汇报自身信息和执行结果。 服务执行和数据存储被分散到不同的服务器集群，服务器集群间通过消息传递进行通信和协调。所有服务器地位平等。
优点 部署结构简单，中央服务器统一管理任务调度，无需考虑对任务的多节点部署，节点之间无需通信和协作， 降低了某一个或者某一簇计算机集群的压力，在解决了单点瓶颈和单点故障问题的同时，还提升了系统的并发度，比较适合大规模集群的管理。
应用 Borg、Kubernetes、Mesos。 Akka、Redis、Cassandra。

调度架构

单体调度 两层调度 共享状态调度
定义 集群中只有一个节点运行调度进程，该节点对其它节点有访问权限，可搜集其它节点资源状态信息以便统一管理。根据用户下发的任务对资源的需求，在调度器中进行任务与资源匹配，再指派给其它节点。 资源的使用状态同时由中央调度器和二层调度器管理。前者负责宏观、大规模资源分配，业务压力较小；后者负责任务与资源匹配，可以有多个、以支持不同任务类型。 将单体调度器分解为多个调度器，每个调度器都有全局资源状态信息，从而实现最优任务调度。
特点 以单个实例的集中式调度器为中心管理资源和调度任务，单体调度器拥有全局资源和全局任务视图，可容易实现对任务的约束并实施全局性调度策略。 解决了单体调度架构中，中央服务器的单点瓶颈问题，且提升了调度效率，支持多种类型任务。 分布式结构，共享集群状态，乐观并发调度，类似数据库中的事务操作。可解决单体调度架构中中央服务器的单点瓶颈问题、解决两层调度架构中任务匹配无法达到全局最优的问题。但存在潜在的资源冲突问题，且实现比较复杂。
应用 Borg、Kubernetes。 Mesos、YARN Google Omega

特点：

  • 规模较大或复杂。

  • 可分解为规模较小、简单的同类问题。

  • 子问题之间相互独立。

  • 子问题的解可合并得到原问题的解。
